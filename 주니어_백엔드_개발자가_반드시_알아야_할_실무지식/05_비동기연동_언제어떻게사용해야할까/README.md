## 비동기 연동
일반적으로 프로그램을 개발할때 실행 순서에 따라서 동기적으로 코드를 작성하는 경우가 많다. 이는 코드를 작성하고 이해하기 쉬우며 디버깅이 용이하다는 장점이 있다. 하지만 외부 시스템 연동 작업시에 동기적으로 코드를 작성하게 되면 몇가지 문제가 발생할 수 있다.
- 연동 작업이 오래 걸릴 경우, 응답성 저하 발생
- 연동 작업이 실패할 경우, 요청이 실패할 가능성 존재
- 트랜잭션 관리의 어려움
이러한 문제를 해결하기 위해서는 비동기적으로 코드를 작성하는 방법을 고려할 수 있다. 비동기적으로 코드를 작성하면, 외부 시스템 연동 작업이 오래 걸리거나 실패하더라도, 프로그램의 응답성을 유지할 수 있으며, 트랜잭션 관리도 용이해진다. <br/>
물론 비동기적으로 코드를 작성하는 것은 동기적으로 코드를 작성하는 것보다 복잡할 수 있으며, 디버깅이 어려울 수 있으며 모니터링 및 로깅이 어려울 수 있다. 때문에 비동기 처리는 상황에 따라서 적절하게 사용하는 것이 중요하다.

### 비동기 연동이 가능한 경우
모든 상황에 비동기 연동이 적합한 것은 아니다. 비동기 연동이 가능한 경우는 다음과 같다.
- 실시간으로 응답을 반환할 필요가 없는 경우
- 실패하더라도 상관이 없는 경우
- 실패하더라도 상관은 없지만 결과적으로 데이터의 일관성을 보장해야하는 경우

#### 실시간으로 응답을 반환할 필요가 없는 경우
실시간으로 응답을 반환할 필요가 없는 작업을 동기적으로 처리하게 될 경우, 외부 연동 응답 시간이 전체 응답 시간에 포함되어 응답 지연이 발생할 수 있다. 이러한 작업은 비동기적으로 분리하여 응답성을 확보하는 것이 좋다. <br/>
예를 들어 회원 가입을 완료한 이후 회원에게 환영 이메일을 보내는 작업은 실시간으로 응답을 반환할 필요가 없다. 오히려 이메일을 전송하는 작업이 오래 걸려 회원 가입 API의 응답이 늦어지는 경우가 발생할 수 있기 때문에, 이메일 전송 작업은 비동기적으로 처리하는 것이 더 효율적일 수 있다.

#### 실패하더라도 상관이 없는 경우
핵심 비즈니스 로직에 영향을 주지 않는 부가 작업의 경우, 실패하더라도 상관이 없는 경우가 많다. 이러한 부가 작업을 핵심 로직과 동기적으로 처리하면, 부가 작업의 실패가 핵심 비즈니스 로직의 실패로 전파될 수 있다. <br/>
예를 들어, 로그나 알림 전송, 통계 수집과 같은 부가 작업이 동기적으로 포함되어 있을 경우, 통계 시스템 장애가 핵심 서비스의 장애로 이어질 수 있다. 이러한 부가 작업은 비동기적으로 분리하여 핵심 로직의 안정성을 보장하는 것이 좋다.

#### 실패하더라도 상관은 없지만 결과적으로 데이터의 일관성을 보장해야하는 경우
일시적인 실패는 허용되지만 결과적으로 데이터의 일관성을 보장해야 하는 경우에도 비동기 연동을 활용할 수 있다. 예를 들어, 검색 인덱스 업데이트나 캐시 갱신, 집계 데이터 업데이트와 같은 작업은 즉시 반영되지 않더라도 일정 시간이 지나면 데이터가 동기화되므로 비즈니스적으로 큰 문제가 발생하지 않는 경우가 많다. 이처럼 **_일시적인 데이터 불일치를 허용하고, 최종적으로 일관성이 보장되는 방식을 결과적 일관성(Eventual Consistency)_** 이라고 한다. <br/>

## 비동기 연동 방식


### 별도의 스레드 사용
별도의 스레드를 사용하여 비동기 작업을 처리하는 방식은 가장 간단한 처리 방법중 하나이다. **_별도의 인프라 없이 애플리케이션 내에서 즉시 적용_** 할 수 있다는 장점이 있다. <br/>
Java 및 Spring에서는 Thread, Runnable, Callable, Future를 활용하여 별도의 스레드를 생성하고 작업을 처리할 수 있으며, 좀더 세밀한 제어가 필요한 경우 CompletableFuture, ExecutorService, @Async 등을 활용하여 비동기 작업을 처리할 수 있다. <br/>
예를 들어, 회원 가입 후 환영 이메일을 보내는 작업을 @Async로 처리하면, 회원 가입 API는 이메일 발송 완료를 기다리지 않고 즉시 응답을 반환할 수 있다.

#### 장점 및 단점

**장점**
- 구현이 간단하고 별도의 인프라가 필요하지 않아 빠르게 적용할 수 있다.

**단점**
- 작업이 메모리에만 존재하기 때문에, 애플리케이션 장애가 발생하는 경우 처리 중인 작업이 유실될 수 있다.
- 서버가 여러 대인 경우, 각 서버의 스레드 풀이 독립적으로 동작하기 때문에 작업 분배가 어렵다.
- 리소스를 공유하기 때문에 비동기 작업이 몰리거나 작업이 오래 걸리는 경우, 전체 시스템의 성능에 영향을 줄 수 있다.

#### 주의해야 할 점

**_1. 스레드 풀 크기 관리_** <br/>
스레드 풀의 크기를 너무 크게 설정하면 리소스가 낭비되고, 너무 작게 설정하면 작업이 큐에 대기하게 되어 처리가 지연될 수 있다. 또한 스레드 풀이 가득 찬 상태에서 큐마저 가득 차면 작업이 거부될 수 있기 때문에, 스레드 풀 크기와 큐 크기를 적절하게 설정하는 것이 중요하다. <br/>
해당 설정은 애플리케이션의 부하 패턴과 작업의 처리 시간에 따라 달라질 수 있으므로, 모니터링을 통해 적절한 크기를 유지하는 것이 좋다. <br/>
만약 작업이 거부되지 않도록 하기위해서는 큐의 크기를 충분히 크게 설정하거나, 작업 거부 정책에서 호출 스레드에서 직접 처리하도록 하는 등의 대체 방안을 고려할 수 있다.

**_2. 작업 유실 가능성_** <br/>
별도의 스레드에서 처리하는 작업은 메모리에만 존재하기 때문에, 애플리케이션이 비정상적으로 종료되면 처리 중이거나 대기 중인 작업이 유실된다. 따라서 유실되어도 비즈니스에 큰 영향이 없는 작업에만 사용하는 것이 좋다. <br/>

**_3. 트랜잭션 전파 안됨_** <br/>
별도의 스레드에서는 기존 스레드의 트랜잭션 컨텍스트를 공유하지 않는다. 따라서 메인 스레드에서 시작한 트랜잭션이 비동기 스레드로 전파되지 않기 때문에, 비동기 작업에서 DB 작업이 필요한 경우 별도의 트랜잭션을 시작해야 한다. <br/>

**_4. 예외 처리_** <br/>
별도의 스레드에서 발생한 예외는 호출자에게 전파되지 않는다. 예외가 발생하더라도 호출자는 이를 인지할 수 없기 때문에, 비동기 작업 내에서 예외를 적절히 처리하고 로깅이나 알람을 통해 문제를 인지할 수 있도록 하는 것이 중요하다. <br/>

> 별도의 스레드를 활용한 비동기 처리는 **_유실되어도 비즈니스에 큰 영향이 없는 작업에만 사용_** 하는 것이 좋다.


### 배치 전송
배치 전송은 **_처리할 데이터를 DB에 저장해 두고, 스케줄러가 주기적으로 조회하여 일괄 처리하는 방식_** 이다. Spring의 @Scheduled, Quartz, Spring Batch 등을 활용할 수 있다. <br/>
일반적으로 상태 필드를 활용하여 처리 상태를 관리한다. 예를 들어, 알림 전송 작업을 배치로 처리한다면, 알림 데이터를 "대기(PENDING)" 상태로 저장한 후, 스케줄러가 주기적으로 "대기" 상태의 데이터를 조회하여 전송하고, 처리 결과에 따라 "완료(COMPLETED)" 또는 "실패(FAILED)"로 상태를 변경한다.

#### 장점 및 단점

**장점**
- 처리해야할 데이터를 DB에 저장하기 때문에 작업 유실 위험이 낮다.
- 대량의 데이터를 효율적으로 처리할 수 있다.
- 작업이 실패하더라도 재시도 로직을 구현하여 안정적으로 처리할 수 있다.

**단점**
- 배치 주기만큼 처리 지연이 발생하기 때문에 실시간성이 떨어진다. 
- 처리할 데이터가 많을 경우 배치 시스템의 부하가 집중될 수 있다.

#### 주의해야 할 점
**_1. 다중 서버 환경에서의 중복 실행_** <br/>
서버가 여러 대인 환경에서 각 서버마다 동일한 스케줄러가 실행되면, 같은 데이터를 여러 서버에서 동시에 처리하여 중복 실행이 발생할 수 있다. 이를 방지하기 위해 **_분산 락(ShedLock 등)_** 을 활용하여 하나의 서버에서만 배치가 실행되도록 제한할 수 있다. <br/>

**_2. 배치 실패 시 재처리_** <br/>
배처 처리 도중 애플리케이션에서 장애 가 발생하게 될 경우 처리 중이던 데이터의 상태가 "처리 중(PROCESSING)"에 머물러 있을 수 있다. 때문에 실패한 시점부터 재처리할 수 있도록, 배치 처리 도중 장애가 발생하면 해당 데이터를 "대기(PENDING)" 상태로 되돌린 후, 배치를 재실행하여 다시 처리할 수 있도록 하는 것이 중요하다. <br/>
이때 첫번째로 중요한 것은 배치가 실패했다는 사실을 인지하는 것이다. 이를 위해 알람을 통해 문제를 인지할 수 있도록 하는 것이 중요하다. 이후 수동 또는 자동으로 해당 데이터를 "대기(PENDING)" 상태로 되돌리는 복구 작업을 수행하여, 다음 배치 실행 시 다시 처리할 수 있도록 해야 한다. <br/>
Spring Batch에서는 정크 단위로 트랜잭션을 관리하기 때문에, 배치 처리 도중 장애가 발생하면 해당 정크 단위의 트랜잭션이 롤백되어 데이터 상태가 "처리 중"에서 "대기"로 자동으로 되돌아간다. 이후 배치 상태를 DB에 저장되어 있기 때문에 실패한 지점부터 재시작할 수 있다.

**_3. 처리량 증가에 따른 배치 시간 증가_** <br/>
처리할 데이터가 늘어나면 배치 실행 시간이 길어질 수 있다. 배치 실행 시간이 배치 주기를 초과하면, 다음 배치와 겹치거나 처리 지연이 누적될 수 있다. 이를 해결하기 위해서는 몇가지 방법을 고려할 수 있다. <br/>
1. 배치 주기를 늘려서 배치 실행 시간이 충분히 확보되도록 하는 방법이 있다.
2. 한번에 처리해야할 데이터의 크기를 정크 단위로 나누어 처리하는 방법이 있다. 이는 읽기 성능을 향상시키고, 장애 발생 시 롤백 범위를 줄이는 효과가 있다. <br/>
3. 하나의 배치 작업에서 처리해야하는 경우가 많은 경우, 멀티 스레드를 활용하여 병렬로 처리하는 방법이 있다. <br/>
4. 여러 대의 서버를 운영하여 배치 작업을 분산 처리하는 방법도 있다. 1번 서버의 경우 1~1000번 데이터를 처리하고, 2번 서버의 경우 1001~2000번 데이터를 처리하는 방식으로 배치 작업을 분산하여 처리할 수 있다. <br/>

### 메시징
메시징은 **_메시지 브로커(Kafka, RabbitMQ, SQS 등)를 통해 프로듀서가 메시지를 발행하고, 컨슈머가 메시지를 소비하여 처리하는 방식_** 이다. 프로듀서와 컨슈머가 직접 통신하지 않고 메시지 브로커를 중간에 두기 때문에, 시스템 간 **_느슨한 결합(Loose Coupling)_** 을 유지할 수 있다. <br/>
예를 들어, 주문 서비스에서 주문이 생성되면 메시지 브로커에 "주문 생성" 메시지를 발행하고, 알림 서비스가 해당 메시지를 소비하여 사용자에게 알림을 보내는 방식으로 처리할 수 있다.

#### 장점 및 단점

**장점**
- 프로듀서와 컨슈머간의 통신이 메시지 브로커를 통해 간접적으로 이루어지기 때문에, 서로의 존재를 알 필요가 없다. 이로 인해 시스템 간 결합도가 낮아지고, 한쪽 시스템을 변경하더라도 다른 쪽에 영향을 주지 않는다.
- 새로운 컨슈머를 추가하더라도 기존 프로듀서의 코드를 수정할 필요가 없기 때문에, 시스템 확장이 용이하다. 
- 메시지 브로커가 메시지를 영속적으로 저장하기 때문에, 컨슈머가 일시적으로 장애가 발생하더라도 복구 후 메시지를 다시 소비할 수 있다. 물론 메시징 시스템 별로 메시지를 보존하지 않는 경우도 있고, 보존하더라도 보존 기간이 있기 때문에, 메시지 보존 정책을 잘 이해하고 설정하는 것이 중요하다.

**단점**
- 메시지 브로커 인프라를 별도로 구축하고 운영해야 하므로 운영 복잡성이 증가한다.
- 메시지 흐름을 추적하기 어려워 모니터링 및 디버깅이 어려울 수 있다.

#### 주의해야 할 점
**_1. 메시지 발행 보장_** <br/>
DB에 데이터를 저장하는 작업과 메시지를 발행하는 작업은 서로 다른 시스템에서 이루어지기 때문에, DB 저장은 성공했지만 메시지 발행에 실패하는 경우가 발생할 수 있다. 이 경우 컨슈머는 해당 이벤트를 인지하지 못하게 된다. <br/>
이를 해결하기 위해 **_트랜잭셔널 아웃박스 패턴_** 을 활용하여 DB 저장과 메시지 발행의 원자성을 보장할 수 있다. <br/>
또한 메시지를 발행하는 시점에 일시적인 장애가 발생할 수 있는데, 이 경우에는 재시도 로직을 구현하여 메시지 발행이 성공할 때까지 재시도하는 방법도 있다.

**_2. 중복 소비_** <br/>
메시지 브로커는 일반적으로 **_최소 한 번 전달(At-Least-Once)_** 을 보장하기 때문에, 네트워크 이슈나 컨슈머 장애 등으로 인해 동일한 메시지가 여러 번 전달될 수 있다. <br/>
이를 해결하기 위해 **_멱등성(Idempotency)_** 처리를 적용하여, 동일한 메시지가 여러 번 처리되더라도 결과가 동일하도록 설계해야 한다. <br/>
애초에 상태 변경 로직이 멱등성이 보장되는 방식으로 설계하는 것이 가장 좋지만, 멱등성이 보장되지 않는 경우에는 메시지에 고유 식별자(ID)를 포함하여, 컨슈머 측에서 현재 처리 중이거나 처리된 메시지 ID를 저장하여 중복 처리를 방지하는 방법도 있다. 대표적으로 Redis NX 옵션을 활용하여, 메시지 ID를 키로 사용하여 중복 처리를 방지할 수 있다. <br/>

**_3. 메시지 순서_** <br/>
메시지 브로커에서 파티션이나 큐 설정에 따라 메시지의 처리 순서가 보장되지 않을 수 있다. 예를 들어, 주문 생성 → 주문 취소 순서로 메시지가 발행되었지만, 컨슈머에서는 주문 취소 → 주문 생성 순서로 처리될 수 있다. 메시지 순서가 중요한 경우에는 동일한 키를 가진 메시지가 동일한 파티션으로 전달되도록 **_파티션 키_** 를 적절하게 설정하여 순서를 보장할 수 있다. <br/>

**_4. 컨슈머측 재시도_** <br/>
컨슈머가 메시지를 처리하는 도중 일시적인 장애가 발생하여 처리가 실패할 수 있다. 이 경우 컨슈머는 메시지 브로커에게 메시지가 처리되었다고 알리지 않는 것이 중요하다. 그래야 메시지 브로커가 컨슈머에게 해당 메시지를 다시 전달하거나, 컨슈머가 메시지 브로커에서 해당 메시지를 다시 소비하여 재시도할 수 있다. <br/>
이때 재시도 횟수나 간격을 적절하게 설정하는 것이 좋고, 재시도 임계치를 초과한 메시지는 DLQ(Dead Letter Queue) 로 이동시켜 별도로 관리하는 것이 좋다. DLQ 에서 관리되는 메시지는 수동으로 문제를 분석하여 재처리하거나 처리하지 않을 수 있다. <br/>


### CDC
CDC(Change Data Capture)는 **_DB의 변경 로그(MySQL binlog, PostgreSQL WAL 등)를 캡처하여 이벤트로 발행하는 방식_** 이다. 대표적으로 Debezium과 Kafka Connect를 조합하여 사용한다. <br/>
CDC는 DB에 커밋된 변경사항을 변경 로그에서 직접 읽어오기 때문에, **_애플리케이션 코드를 변경하지 않고도 데이터 변경 이벤트를 발행_** 할 수 있다. 또한 DB 트랜잭션이 커밋된 데이터만 캡처하기 때문에, 메시지 발행의 신뢰성이 높다.

#### 장점 및 단점

**장점**
- 애플리케이션 코드 변경이 최소화되며, DB 트랜잭션과 자연스럽게 통합되어 커밋된 데이터만 캡처하므로 메시지 발행 보장이 용이하다.
- 별도의 아웃박스 테이블 없이도 안정적인 이벤트 발행이 가능하다.

**단점**
- CDC 서버와 메시지 브로커를 함께 운영해야 하므로 인프라 복잡성이 높다.
- DB의 변경 로그에 의존하기 때문에 특정 DB에 종속적이며, 스키마 변경에 민감하다.

#### 주의해야 할 점
**_1. DB 변경 로그 관리_** <br/>
CDC를 사용하려면 DB의 변경 로그(binlog, WAL 등)가 활성화되어 있어야 한다. 변경 로그는 저장 공간을 차지하기 때문에, 보관 주기와 저장 공간을 적절하게 관리해야 한다. <br/>

**_2. 스키마 변경의 영향_** <br/>
DB 스키마가 변경되면 CDC 파이프라인에서 캡처하는 이벤트의 구조도 변경된다. 이 경우 컨슈머가 변경된 구조를 처리하지 못하면 오류가 발생할 수 있다. 이를 해결하기 위해 **_스키마 레지스트리_** 를 활용하여 이벤트 구조의 호환성을 관리하고, 하위 호환성을 유지하는 방향으로 스키마를 변경하는 것이 좋다. <br/>

**_3. CDC 서버 장애 시 이벤트 누락_** <br/>
CDC 서버에 장애가 발생하면, 장애 기간 동안의 변경 이벤트가 누락될 수 있다. 장애 복구 시 변경 로그에서 이어서 읽어올 수 있지만, 변경 로그의 보관 주기가 지났다면 해당 이벤트를 복구할 수 없다. 이를 방지하기 위해 모니터링 및 이중화를 통해 장애를 조기에 감지하고, **_변경 로그의 보관 주기를 CDC 서버의 장애 복구 시간보다 충분히 길게 설정_** 하는 것이 좋다. <br/>
