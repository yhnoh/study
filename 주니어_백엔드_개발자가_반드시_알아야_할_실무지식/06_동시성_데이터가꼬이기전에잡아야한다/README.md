## 동시성 이슈
동시성 이슈란 여러개의 스레드가 동시에 같은 자원에 접근할 때 발생하는 문제다. 이때 자원은 메모리, 파일, 데이터베이스 등 다양할 수 있다. 문제는 여러 스레드가 동시에 자원에 접근할 때, 예상치 못한 결과가 발생할 수 있다는 것이다. <br/>
이로 인해 데이터의 무결성이 깨지게 되며 더욱 큰 문제는 이를 재현하는 것이 어렵고 디버깅이 어려우며 이로 인해서 문제를 해결하는 데 많은 시간과 노력이 필요하다는 것이다. <br/>
때문에 멀티 스레드 환경에서 프로그래밍을 할 때는 초기에 동시성 이슈를 고려하여 설계하는 것이 중요하다. 이를 위해서는 스레드 간의 자원 접근을 제어하는 방법을 사용해야 한다. <br/>

### 동시성 이슈가 발생하는 상황
동시성 이슈의 대표적인 원인은 **_경쟁 조건(Race Condition)_** 이다. 경쟁 조건이란 두 개 이상의 스레드가 공유 자원에 동시에 접근하여 읽기-수정-쓰기(Read-Modify-Write) 작업을 수행할 때, 실행 순서에 따라 결과가 달라지는 현상을 말한다. <br/>
예를 들어, 상품의 재고가 1개 남은 상황에서 두 명의 사용자가 동시에 주문을 요청하는 경우를 생각해보자. 정상적이라면 한 명만 주문에 성공하고 나머지 한 명은 재고 부족으로 주문이 실패해야 한다. 하지만 두 스레드가 동시에 재고를 조회하면 둘 다 재고가 1개 있다고 판단하여, 두 주문 모두 성공하게 되고 재고는 -1이 되는 문제가 발생할 수 있다. <br/>

| 순서 | 스레드 A (사용자 1) | 스레드 B (사용자 2) | 재고 |
|------|---------------------|---------------------|------|
| 1 | 재고 조회 → 1개 | | 1 |
| 2 | | 재고 조회 → 1개 | 1 |
| 3 | 재고 차감 → 0개 | | 0 |
| 4 | | 재고 차감 → -1개 | -1 |

이처럼 동시성 이슈는 단일 스레드에서는 발생하지 않지만, 멀티 스레드 환경에서 공유 자원에 대한 접근이 제어되지 않을 때 발생한다. 특히 웹 애플리케이션에서는 여러 사용자의 요청이 동시에 처리되기 때문에, 이러한 문제가 빈번하게 발생할 수 있다. <br/>

### 동기화 메커니즘
동시성 이슈를 해결하기 위해서는 공유 자원에 대한 접근을 제어하는 동기화 메커니즘을 사용해야 한다. 동기화 메커니즘의 핵심 개념은 **_상호 배제(Mutual Exclusion)_** 로, 한 스레드가 공유 자원을 사용하는 동안 다른 스레드가 해당 자원에 접근하지 못하도록 제한하는 것이다. <br/>

#### 뮤텍스(Mutex)
뮤텍스는 **_하나의 스레드만 공유 자원에 접근할 수 있도록 제한하는 동기화 메커니즘_** 이다. 스레드가 공유 자원에 접근하려면 먼저 뮤텍스의 락(Lock)을 획득해야 하며, 작업이 끝나면 락을 해제한다. 락을 획득하지 못한 다른 스레드는 락이 해제될 때까지 대기한다.

| 레이어 | 구현 | 설명 |
|--------|------|------|
| 애플리케이션 | `synchronized`, `ReentrantLock` | 한 번에 하나의 스레드만 임계 구역 진입 허용 |
| DB | `SELECT ... FOR UPDATE` | 트랜잭션 종료 시까지 해당 행에 배타적 락 획득 |

뮤텍스는 가장 기본적인 동기화 메커니즘으로, 단일 스레드만 자원에 접근할 수 있도록 보장하지만, 동시에 여러 스레드가 읽기 작업을 수행하는 경우에도 락이 필요하기 때문에 처리량이 저하될 수 있다. <br/>
주로 데이터의 정합성이 매우 중요하고, 충돌이 빈번하게 발생하는 상황에서 사용된다. 예를 들어, 재고 차감, 계좌 이체 등과 같이 동일한 데이터에 대한 동시 수정이 빈번한 경우에 적합하다. <br/>

#### 세마포어(Semaphore)
세마포어는 **_동시에 공유 자원에 접근할 수 있는 스레드의 수를 제한하는 동기화 메커니즘_** 이다. 뮤텍스가 한 번에 하나의 스레드만 허용하는 반면, 세마포어는 지정된 수만큼의 스레드가 동시에 접근할 수 있다.

| 레이어 | 구현 | 설명 |
|--------|------|------|
| 애플리케이션 | `Semaphore` | 동시 접근 가능 스레드 수 제한, 커넥션 풀 구현 등에 활용 |
| DB | `max_connections`, 커넥션 풀 | 동시 접속 가능한 커넥션 수를 제한하여 자원 초과 방지 |

세마포어는 뮤텍스보다 유연한 동기화 메커니즘으로, 동시에 여러 스레드가 임계 구역에 진입할 수 있도록 허용하지만, 여전히 자원에 대한 과도한 접근을 방지할 수 있다. <br/>
주로 커넥션 풀이나 대기열 같이 제한된 자원에 대한 동시 접근을 제어하는 데 사용된다. <br/>


#### 비관적 락(Pessimistic Lock)
비관적 락은 **_데이터에 접근할 때 충돌이 발생할 것이라고 가정하고, 먼저 락을 획득한 후 작업을 수행하는 방식_** 이다. 락을 먼저 확보하기 때문에 충돌 시 재시도 없이 순차적으로 처리된다. <br/>
비관적 락은 락의 종류에 따라 배타적 락과 공유 락으로 나눌 수 있다.

| 레이어 | 구현 | 설명 |
|--------|------|------|
| 애플리케이션 | `synchronized`, `ReentrantLock` | 작업 전 락을 획득하고 완료 후 해제 |
| DB | `SELECT ... FOR UPDATE`, `UNIQUE` | 조회 시점에 배타적 락 획득, 트랜잭션 종료까지 유지 |

뮤텍스와 비슷하게 단일 스레드만 자원에 접근할 수 있도록 보장하지만, 동시에 여러 스레드가 읽기 작업을 수행하는 경우에도 락이 필요하기 때문에 처리량이 저하될 수 있다. <br/>

#### 낙관적 락(Optimistic Lock)
낙관적 락은 **_데이터에 접근할 때 충돌이 발생하지 않을 것이라고 가정하고, 실제로 데이터를 수정하는 시점에 충돌 여부를 확인하는 방식_** 이다. 실제 락을 걸지 않고, 수정 시점에 데이터 변경 여부를 확인하여 충돌을 감지한다.


| 레이어 | 구현 | 설명 |
|--------|------|------|
| 애플리케이션 | `AtomicInteger`, `compareAndSet()`, JPA `@Version` | CAS 연산 또는 버전 비교로 충돌 감지 후 재시도 |
| DB | version 컬럼 + 조건부 UPDATE | 조회 시 버전을 읽고, 수정 시 버전 일치 여부 확인 |

낙관적 락은 충돌이 드문 환경에서 유리하며, 실제로 DB 락을 걸지 않기 때문에 대기가 발생하지 않아 처리량이 높다. 그러나 충돌이 발생하면 재시도 로직을 직접 구현해야 하며, 충돌 빈도가 높은 환경에서는 재시도로 인하여 비관적 락보다 성능이 저하될 수 있다. <br/>

> CAS(Compare-And-Swap) 연산은 낙관적 락의 한 형태로, 메모리 위치의 값을 읽고, 연산을 수행한 이후, 예상한 값과 일치할 때만 새로운 값으로 업데이트하는 원자적 연산이다. <br/>
> 만약 연산을 수행한 이후 예상한 값과 일치하지 않으면, 다른 스레드가 해당 값을 변경했다는 의미이므로, 연산을 재시도 할 수 있는데, 이를 통해 락 없이도 동시성 제어가 가능하다.

#### 읽기 쓰기 락(Read-Write Lock)
읽기 쓰기 락은 **_읽기 작업은 여러 스레드가 동시에 수행할 수 있지만, 쓰기 작업은 한 번에 하나의 스레드만 수행할 수 있도록 제한하는 동기화 메커니즘_** 이다. 뮤텍스가 읽기/쓰기 구분 없이 하나의 스레드만 접근을 허용하는 것과 달리, 읽기 쓰기 락은 읽기 작업의 병렬 처리를 허용해 처리량을 높일 수 있다. <br/>
읽기 쓰기 락은 배타적 락(쓰기 락)과 공유 락(읽기 락)으로 구성된다. <br/>
주로 데이터에 대한 읽기 작업이 빈번하고, 쓰기 작업이 상대적으로 드문 경우에 사용된다. 예를 들어, 상품 목록 조회는 빈번하지만, 상품 정보 수정은 드문 경우에 적합하다. <br/>


#### 배타적 락/쓰기 락(Exclusive Lock/Write Lock)
배타적 락은 **_한 스레드(또는 트랜잭션)가 락을 획득하면, 다른 스레드는 해당 데이터를 읽거나 수정할 수 없도록 제한하는 락_** 이다. 데이터를 수정하는 작업에 주로 사용되며, 락이 걸린 데이터에 대해 다른 스레드가 읽기 또는 쓰기를 시도하면 락이 해제될 때까지 대기해야 한다.

| 레이어 | 구현 | 설명 |
|--------|------|------|
| 애플리케이션 | `lock.writeLock().lock()` | `ReentrantReadWriteLock`의 쓰기 락 |
| DB | `SELECT ... FOR UPDATE` | 트랜잭션 종료 시까지 해당 행에 배타적 락 획득 |


#### 공유 락/읽기 락(Shared Lock/Read Lock)
공유 락은 **_여러 스레드(또는 트랜잭션)가 동시에 데이터를 읽을 수 있지만, 수정은 할 수 없도록 제한하는 락_** 이다. 주로 데이터를 읽는 동안 다른 스레드가 수정하는 것을 방지하고 싶을 때 사용하며, 쓰기를 시도하는 스레드는 공유 락이 해제될 때까지 대기해야 한다.

| 레이어 | 구현 | 설명 |
|--------|------|------|
| 애플리케이션 | `lock.readLock().lock()` | `ReentrantReadWriteLock`의 읽기 락 |
| DB | `SELECT ... FOR SHARE` | 트랜잭션 종료 시까지 해당 행에 공유 락 획득 |


### 동기화 메커니즘 사용시 주의사항

#### 1. 락 해제하기(Deadlock)
동기화 메커니즘을 사용할 때 가장 중요한 것은 락을 반드시 해제하는 것이다. 락이 해제되지 않으면 다른 스레드가 해당 자원에 접근할 수 없게 되어, 로직을 수행할 수 없다. 때문에 락을 획득한 후에는 반드시 락을 해제하는 코드를 작성해야 한다. <br/>

#### 2. 락 대기 시간(Lock Timeout) 설정하기
동기화 메커니즘을 사용할 때, 락 대기 시간을 설정하는 것이 중요하다. 락 대기 시간이 너무 길어지면, 다른 스레드가 해당 자원에 접근할 수 없게 되어, 전체 시스템의 성능이 저하될 수 있다. 따라서 락 대기 시간을 적절히 설정하여, 일정 시간 내에 락을 획득하지 못하면 실패 처리하는 것이 좋다. <br/>

#### 3. 락 획득 및 해제 순서 일관성 유지하기
만약 하나의 로직에서 여러개의 락을 획득해야하는 경우, 락을 획득하는 순서와 해제하는 순서를 일관되게 유지하는 것이 중요하다. 그렇지 않으면 동시성 이슈가 발생할 수 있다. <br/>
예를 들어 하나의 로직에 자원 A와 자원 B에 대하여 순차적으로 락을 획득한다고 가정해보자.
- Thread-1: 자원 A 락 획득 -> 자원 B 락 획득
- Thread-1: 자원 A 락 해제 -> 자원 B 락 해제중
- Thread-2: 자원 A 락 획득 <br/>

위와 같이 아직 Thread-1의 로직은 종료되지 않았지만, Thread-2가 자원 A의 락을 획득하여 Thread-1이 자원 B의 락을 해제하기 전에 자원 A에 접근할 수 있게 되며 이로 인해 동시성 이슈가 발생할 수 있다. <br/>
때문에 락을 획득하는 순서와 해제하는 순서를 일관되게 유지하기 위하여 Stack 구조로 락을 획득하고 해제하는 것이 좋다. <br/>

#### 4. 데드락 주의하기
데드락이란 두 개 이상의 스레드가 서로가 보유한 락을 기다리면서 무한 대기에 빠지는 상황을 말한다. 이는 동시성 이슈 중에서도 가장 심각한 문제 중 하나로, 시스템이 멈추거나 응답하지 않게 될 수 있으며 이를 해결하기 위해서는 시스템을 재시작해야 하는 경우도 있다. <br/>
예를 들어, Thread-1이 자원 A의 락을 획득한 후 자원 B의 락을 요청하고, Thread-2가 자원 B의 락을 획득한 후 자원 A의 락을 요청하면 서로 무한 대기에 빠진다.
- Thread-1: 자원 A 락 획득 -> 자원 B 락 요청
- Thread-2: 자원 B 락 획득 -> 자원 A 락 요청 <br/>

#### 5. 락의 범위 최소화하기
동시성 이슈를 해결하기 위하여 락을 사용하기는 하지만, 락은 성능 저하의 원인이 될 수 있기 때문에, 락의 범위를 최소화하는 것이 중요하다. <br/>
락이란 결국 다른 스레드가 해당 자원에 접근할 수 없도록 제한하는 것이기 때문에, 그만큼 응답 지연이 발생할 수 밖에 없다. 때문에 락이 필요한 최소한의 코드 블록에만 락을 적용하고, 락이 필요하지 않은 부분에서는 락을 해제하여 다른 스레드가 해당 자원에 접근할 수 있도록 하는 것이 좋다. <br/>
