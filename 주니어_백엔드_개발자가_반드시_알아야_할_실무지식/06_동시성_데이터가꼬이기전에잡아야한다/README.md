## 동시성 이슈
동시성 이슈란 여러개의 스레드가 동시에 같은 자원에 접근할 때 발생하는 문제다. 이때 자원은 메모리, 파일, 데이터베이스 등 다양할 수 있다. 문제는 여러 스레드가 동시에 자원에 접근할 때, 예상치 못한 결과가 발생할 수 있다는 것이다. <br/>
이로 인해 데이터의 무결성이 깨지게 되며 더욱 큰 문제는 이를 재현하는 것이 어렵고 디버깅이 어려우며 이로 인해서 문제를 해결하는 데 많은 시간과 노력이 필요하다는 것이다. <br/>
때문에 멀티 스레드 환경에서 프로그래밍을 할 때는 초기에 동시성 이슈를 고려하여 설계하는 것이 중요하다. 이를 위해서는 스레드 간의 자원 접근을 제어하는 방법을 사용해야 한다. <br/>

### 동시성 이슈가 발생하는 상황
동시성 이슈의 대표적인 원인은 **_경쟁 조건(Race Condition)_** 이다. 경쟁 조건이란 두 개 이상의 스레드가 공유 자원에 동시에 접근하여 읽기-수정-쓰기(Read-Modify-Write) 작업을 수행할 때, 실행 순서에 따라 결과가 달라지는 현상을 말한다. <br/>
예를 들어, 상품의 재고가 1개 남은 상황에서 두 명의 사용자가 동시에 주문을 요청하는 경우를 생각해보자. 정상적이라면 한 명만 주문에 성공하고 나머지 한 명은 재고 부족으로 주문이 실패해야 한다. 하지만 두 스레드가 동시에 재고를 조회하면 둘 다 재고가 1개 있다고 판단하여, 두 주문 모두 성공하게 되고 재고는 -1이 되는 문제가 발생할 수 있다. <br/>

| 순서 | 스레드 A (사용자 1) | 스레드 B (사용자 2) | 재고 |
|------|---------------------|---------------------|------|
| 1 | 재고 조회 → 1개 | | 1 |
| 2 | | 재고 조회 → 1개 | 1 |
| 3 | 재고 차감 → 0개 | | 0 |
| 4 | | 재고 차감 → -1개 | -1 |

이처럼 동시성 이슈는 단일 스레드에서는 발생하지 않지만, 멀티 스레드 환경에서 공유 자원에 대한 접근이 제어되지 않을 때 발생한다. 특히 웹 애플리케이션에서는 여러 사용자의 요청이 동시에 처리되기 때문에, 이러한 문제가 빈번하게 발생할 수 있다. <br/>

### 동기화 메커니즘
동시성 이슈를 해결하기 위해서는 공유 자원에 대한 접근을 제어하는 동기화 메커니즘을 사용해야 한다. 동기화 메커니즘의 핵심 개념은 **_상호 배제(Mutual Exclusion)_** 로, 한 스레드가 공유 자원을 사용하는 동안 다른 스레드가 해당 자원에 접근하지 못하도록 제한하는 것이다. <br/>

#### 뮤텍스(Mutex)
뮤텍스는 **_하나의 스레드만 공유 자원에 접근할 수 있도록 제한하는 동기화 메커니즘_** 이다. 스레드가 공유 자원에 접근하려면 먼저 뮤텍스의 락(Lock)을 획득해야 하며, 작업이 끝나면 락을 해제한다. 락을 획득하지 못한 다른 스레드는 락이 해제될 때까지 대기한다. <br/>
Java에서는 `synchronized` 키워드나 `ReentrantLock` 클래스가 뮤텍스 역할을 한다.

#### 세마포어(Semaphore)
세마포어는 **_동시에 공유 자원에 접근할 수 있는 스레드의 수를 제한하는 동기화 메커니즘_** 이다. 뮤텍스가 한 번에 하나의 스레드만 허용하는 반면, 세마포어는 지정된 수만큼의 스레드가 동시에 접근할 수 있다. <br/>
예를 들어, 커넥션 풀에서 동시에 사용할 수 있는 커넥션 수를 제한하는 것이 세마포어의 대표적인 활용 예시다. Java에서는 `Semaphore` 클래스를 통해 사용할 수 있다.

#### 비관적 락(Pessimistic Lock)
비관적 락은 **_데이터에 접근할 때 충돌이 발생할 것이라고 가정하고, 먼저 락을 획득한 후 작업을 수행하는 방식_** 이다. 주로 DB에서 `SELECT ... FOR UPDATE` 구문을 통해 특정 행에 대한 락을 획득하여, 다른 트랜잭션이 해당 행을 수정하지 못하도록 제한한다. <br/>
비관적 락은 락의 종류에 따라 배타적 락과 공유 락으로 나눌 수 있다.

#### 낙관적 락(Optimistic Lock)
낙관적 락은 **_데이터에 접근할 때 충돌이 발생하지 않을 것이라고 가정하고, 실제로 데이터를 수정하는 시점에 충돌 여부를 확인하는 방식_** 이다. 일반적으로 버전(version) 컬럼을 활용하여, 데이터를 읽을 때의 버전과 수정할 때의 버전이 일치하는지 비교한다. 버전이 일치하지 않으면 다른 트랜잭션이 먼저 수정한 것이므로 충돌로 판단하고 작업을 실패 처리한다. <br/>

```sql
-- 데이터 조회 시 버전 함께 조회
SELECT id, stock, version FROM product WHERE id = 1;
-- stock: 10, version: 1

-- 데이터 수정 시 버전 비교
UPDATE product SET stock = 9, version = 2 WHERE id = 1 AND version = 1;
-- 다른 트랜잭션이 먼저 수정했다면 version이 이미 2로 변경되어 있으므로 업데이트 대상이 0건이 된다.
```

JPA에서는 `@Version` 어노테이션을 통해 낙관적 락을 쉽게 적용할 수 있다. 충돌이 발생하면 `OptimisticLockException`이 발생한다. <br/>

#### 배타적 락/쓰기 락(Exclusive Lock/Write Lock)
배타적 락은 **_한 트랜잭션이 락을 획득하면, 다른 트랜잭션은 해당 데이터를 읽거나 수정할 수 없도록 제한하는 락_** 이다. 데이터를 수정하는 작업에 주로 사용되며, DB에서는 `SELECT ... FOR UPDATE` 구문을 통해 획득할 수 있다. <br/>
배타적 락이 걸린 데이터에 대해 다른 트랜잭션이 읽기 또는 쓰기를 시도하면, 락이 해제될 때까지 대기해야 한다.

#### 공유 락/읽기 락(Shared Lock/Read Lock)
공유 락은 **_여러 트랜잭션이 동시에 데이터를 읽을 수 있지만, 수정은 할 수 없도록 제한하는 락_** 이다. DB에서는 `SELECT ... FOR SHARE` 또는 `SELECT ... LOCK IN SHARE MODE` 구문을 통해 획득할 수 있다. <br/>
공유 락이 걸린 데이터에 대해 다른 트랜잭션은 읽기는 가능하지만, 쓰기를 시도하면 공유 락이 해제될 때까지 대기해야 한다. 주로 데이터를 읽는 동안 다른 트랜잭션이 수정하는 것을 방지하고 싶을 때 사용한다.

## 애플리케이션 동시성 제어하기
애플리케이션 레벨에서의 동시성 제어는 Java에서 제공하는 동기화 도구를 활용하여 공유 자원에 대한 접근을 제어하는 방식이다. 대표적으로 `synchronized` 키워드와 `ReentrantLock` 클래스가 있다. <br/>

`synchronized`는 메서드나 블록 단위로 락을 걸어, 한 번에 하나의 스레드만 해당 코드를 실행할 수 있도록 한다. `ReentrantLock`은 `synchronized`보다 세밀한 제어가 가능하며, 타임아웃 설정이나 공정성(fairness) 설정 등을 지원한다. <br/>

```java
// synchronized 사용
public synchronized void decreaseStock(Long productId) {
    Product product = productRepository.findById(productId);
    product.decreaseStock();
    productRepository.save(product);
}

// ReentrantLock 사용
private final ReentrantLock lock = new ReentrantLock();

public void decreaseStock(Long productId) {
    lock.lock();
    try {
        Product product = productRepository.findById(productId);
        product.decreaseStock();
        productRepository.save(product);
    } finally {
        lock.unlock();
    }
}
```

#### 장점 및 단점

**장점**
- 별도의 인프라 없이 애플리케이션 코드만으로 동시성을 제어할 수 있다.
- 구현이 비교적 간단하다.

**단점**
- **_단일 JVM(서버) 내에서만 유효_** 하다. 서버가 여러 대인 경우, 각 서버의 락은 독립적으로 동작하기 때문에 동시성 제어가 보장되지 않는다.
- 락을 획득한 스레드가 작업을 완료할 때까지 다른 스레드가 대기해야 하므로, 처리량(throughput)이 저하될 수 있다.

#### 주의해야 할 점

**_1. 단일 서버 환경에서만 유효_** <br/>
애플리케이션 레벨의 동기화는 같은 JVM 내의 스레드 간에만 유효하다. 서버가 여러 대로 운영되는 환경에서는 각 서버가 독립적인 JVM을 사용하기 때문에, `synchronized`나 `ReentrantLock`으로는 서버 간의 동시성을 제어할 수 없다. 이 경우 DB 레벨의 락을 사용해야 한다. <br/>

**_2. 데드락(Deadlock) 주의_** <br/>
두 개 이상의 스레드가 서로 상대방이 보유한 락을 기다리며 무한 대기하는 데드락이 발생할 수 있다. 이를 예방하기 위해 락 획득 순서를 일정하게 유지하거나, 타임아웃을 설정하여 일정 시간 내에 락을 획득하지 못하면 포기하도록 하는 것이 좋다. <br/>

**_3. 트랜잭션과 락의 범위 불일치_** <br/>
Spring에서 `@Transactional`과 `synchronized`를 함께 사용할 때 주의가 필요하다. `synchronized`는 메서드 실행이 끝나면 락을 해제하지만, 트랜잭션의 커밋은 그 이후에 이루어진다. 락이 해제된 시점과 트랜잭션이 커밋된 시점 사이에 다른 스레드가 접근하면, 아직 커밋되지 않은 데이터를 읽을 수 있다. 이를 방지하기 위해 `synchronized` 블록이 트랜잭션 범위를 포함하도록 설계하거나, 락 해제 시점과 커밋 시점이 일치하도록 해야 한다. <br/>

## DB 동시성 제어하기
DB 레벨의 동시성 제어는 DB가 제공하는 락 메커니즘을 활용하여 데이터의 정합성을 보장하는 방식이다. 애플리케이션 레벨과 달리 **_서버가 여러 대인 환경에서도 동시성 제어가 가능_** 하다는 장점이 있다. 여러 서버가 동일한 DB에 접근하더라도, DB가 중앙에서 락을 관리하기 때문이다. <br/>

### 비관적 락 적용

비관적 락은 데이터 충돌이 빈번하게 발생하는 환경에서 유리하다. 락을 먼저 획득한 후 작업을 수행하기 때문에, 충돌 시 재시도 없이 순차적으로 처리된다. <br/>

```sql
-- 비관적 락을 사용한 재고 차감
BEGIN;
SELECT stock FROM product WHERE id = 1 FOR UPDATE; -- 배타적 락 획득
UPDATE product SET stock = stock - 1 WHERE id = 1;
COMMIT; -- 락 해제
```

JPA에서는 `@Lock(LockModeType.PESSIMISTIC_WRITE)` 어노테이션을 통해 비관적 락을 적용할 수 있다. <br/>

#### 장점 및 단점

**장점**
- 데이터 충돌을 원천적으로 방지하므로 데이터 정합성이 강하게 보장된다.
- 충돌 발생 시 재시도 로직이 필요하지 않아 구현이 비교적 단순하다.

**단점**
- 락을 획득한 트랜잭션이 완료될 때까지 다른 트랜잭션이 대기해야 하므로 처리량이 저하될 수 있다.
- 트랜잭션 시간이 길어지면 대기 시간이 늘어나고, 최악의 경우 데드락이 발생할 수 있다.

#### 주의해야 할 점

**_1. 락 대기 시간(Lock Timeout) 설정_** <br/>
비관적 락을 사용할 때 다른 트랜잭션이 이미 락을 보유하고 있으면 대기하게 된다. 대기 시간이 길어지면 사용자 경험이 저하되고 커넥션 풀이 고갈될 수 있으므로, 락 대기 시간(Lock Timeout)을 적절히 설정하여 일정 시간 내에 락을 획득하지 못하면 실패 처리하는 것이 좋다. <br/>

**_2. 데드락(Deadlock) 주의_** <br/>
두 트랜잭션이 서로 다른 순서로 락을 획득하려 할 때 데드락이 발생할 수 있다. 예를 들어, 트랜잭션 A가 상품 1의 락을 획득한 후 상품 2의 락을 요청하고, 트랜잭션 B가 상품 2의 락을 획득한 후 상품 1의 락을 요청하면 서로 무한 대기에 빠진다. 이를 방지하기 위해 락을 획득하는 순서를 일관되게 유지하는 것이 중요하다. <br/>

**_3. 트랜잭션 범위 최소화_** <br/>
비관적 락은 트랜잭션이 종료될 때 해제된다. 트랜잭션 범위가 넓으면 락을 보유하는 시간이 길어져 다른 트랜잭션의 대기 시간이 증가한다. 따라서 락이 필요한 작업만 트랜잭션에 포함하고, 불필요한 작업(외부 API 호출, 파일 처리 등)은 트랜잭션 밖에서 처리하여 락 보유 시간을 최소화하는 것이 좋다. <br/>

### 낙관적 락 적용

낙관적 락은 데이터 충돌이 드물게 발생하는 환경에서 유리하다. 실제 충돌이 발생할 때만 재시도를 하기 때문에, 대부분의 요청이 대기 없이 처리된다. <br/>

```java
// JPA에서 낙관적 락 적용
@Entity
public class Product {
    @Id
    private Long id;
    private int stock;

    @Version
    private Long version;
}
```

충돌이 발생하면 `OptimisticLockException`이 발생하므로, 이를 적절히 처리하는 재시도 로직이 필요하다. <br/>

#### 장점 및 단점

**장점**
- 실제로 DB 락을 걸지 않기 때문에 대기가 발생하지 않아, 충돌이 적은 환경에서 처리량이 높다.
- 데드락이 발생하지 않는다.

**단점**
- 충돌이 발생하면 재시도 로직을 직접 구현해야 한다.
- 충돌 빈도가 높은 환경에서는 재시도가 빈번하게 발생하여 오히려 비관적 락보다 성능이 떨어질 수 있다.

#### 주의해야 할 점

**_1. 재시도 로직 구현_** <br/>
낙관적 락에서 충돌이 발생하면 `OptimisticLockException`이 발생하며, 이를 처리하는 재시도 로직이 필요하다. 재시도 횟수와 간격을 적절하게 설정하고, 재시도 횟수를 초과하면 사용자에게 실패를 알리는 처리가 필요하다. <br/>

**_2. 충돌 빈도가 높으면 비효율적_** <br/>
낙관적 락은 충돌이 적은 환경에서 유리하다. 동일한 데이터에 대한 동시 수정이 빈번한 경우, 재시도가 반복되면서 오히려 성능이 저하될 수 있다. 이 경우 비관적 락을 사용하는 것이 더 효율적이다. <br/>

### 비관적 락 vs 낙관적 락 선택 기준

비관적 락과 낙관적 락은 절대적으로 하나가 우월한 것이 아니며, 상황에 따라 적합한 방식을 선택해야 한다. <br/>

| 기준 | 비관적 락 | 낙관적 락 |
|------|-----------|-----------|
| 충돌 빈도 | 충돌이 잦은 경우 유리 | 충돌이 드문 경우 유리 |
| 처리량 | 락 대기로 인해 처리량이 낮을 수 있음 | 대기가 없어 처리량이 높음 |
| 데이터 정합성 | 강하게 보장 | 충돌 시 재시도로 보장 |
| 구현 복잡도 | 재시도 불필요, 비교적 단순 | 재시도 로직 필요 |
| 데드락 가능성 | 있음 | 없음 |

예를 들어, 재고 차감처럼 동일한 상품에 대한 동시 주문이 빈번한 경우에는 비관적 락이 적합하다. 반면, 사용자 프로필 수정처럼 동일한 데이터를 동시에 수정하는 경우가 드문 경우에는 낙관적 락이 적합하다. <br/>
